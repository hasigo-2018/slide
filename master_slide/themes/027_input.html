Section 1

なぜ今、Figma Make / Sitesなのか

従来のデザイン→公開フロー

デザイナーの意図が開発者に正確に伝わらない「デザイン債務」問題

Figma Make / Sitesが実現する新フロー

Config 2025で発表された「デザインから公開まで一気通貫」の思想

なぜFigmaがこの機能を作ったのか

業界の3つの変化

  • AIのコード生成能力向上：自然言語からコードへの変換が実用レベルに
  • ノーコード/ローコードの普及：Webflow、Framerなど競合の台頭
  • デザイナーの役割変化：「作る」から「評価・判断・ディレクション」へ

Figmaの戦略的意図

「デザインツール」から「プロダクト開発プラットフォーム」への進化。
デザインから公開まで、Figmaエコシステム内で完結させる狙い。

💡 Visual Architectの視点：AIがコードを書く時代、人間は「何を作るべきか」「これで良いか」を判断する能力が重要

Section 2

Figma Make の仕組み

Figma Make
Prompt-to-App

テキストや既存デザインから、動くプロトタイプ・Webアプリを生成するAI機能

Figma Makeの動作メカニズム

Figma Makeの3つの特徴

1. 反復的な対話

一度で完成させる必要なし。AIと会話しながら「もう少し大きく」「色を変えて」と調整可能。

2. Point & Edit

プロトタイプの特定部分をクリックして、その要素だけをピンポイントで修正指示できる。

3. デザインシステム連携

既存のFigmaライブラリ（カラー、タイポグラフィ）をインポートして、ブランド一貫性を保てる。

⚠️ よくある間違い：最初から完璧なプロンプトを書こうとする
✅ 正解：大まかに始めて、段階的に詳細を詰める

Figma Makeが向いている用途

✅ 向いている

  • アイデア検証：「こんな機能どうだろう？」を即座にプロトタイプ化
  • ステークホルダー説得：静止画より動くものの方が説得力がある
  • 開発者との認識合わせ：実際のコードで実現可能性を示す
  • ユーザーテスト用プロトタイプ：リアルなインタラクションで検証

❌ 向いていない

  • 本番環境のコード生成：保守性・拡張性の観点で不十分
  • 複雑なビジネスロジック：認証、決済、データベース連携など
  • 大規模アプリケーション：900プロンプト超えるとAIが混乱し始める

Section 3

Figma Sites の仕組み

Figma Sites
Design-to-Web

Figmaのデザインをそのまま本物のWebサイトとして公開できる機能

Figma Sitesの動作メカニズム

Figma Sitesの主要機能

1. レスポンシブデザイン

Desktop / Tablet / Mobile のブレークポイントを視覚的に設定。Auto Layout 2.0で自動調整。

2. プリセットインタラクション

ホバー効果、スクロールパララックス、マーキー、カスタムカーソルなど、コード不要で追加。

3. Code Layers（コードレイヤー）

プリセットで足りない場合、AIにプロンプトでカスタムアニメーションを依頼。
React + TypeScript + Tailwind CSSで生成。

4. ワンクリック公開

figma.siteドメインに即座に公開。有料プランでカスタムドメイン接続可能。

Section 4

限界と注意点を知る

Figma Make / Sites の現在の制限事項

⚠️ Figma Make

  • AIクレジット制限：使用量に上限あり（Full Seatで無制限だが将来有料化予定）
  • コンテキスト制限：900プロンプト超えると精度低下・ハルシネーション発生
  • 本番コード品質：保守性の観点で、そのまま本番環境には使いにくい

⚠️ Figma Sites

  • CMS未実装：コンテンツ管理機能はまだベータ（2025年末予定）
  • Eコマース非対応：カート、決済、在庫管理などの機能なし
  • フォーム制限：サードパーティ連携なし（問い合わせフォームの自由度低い）
  • 多言語対応限定的：国際化サイトには不向き

Section 5

評価・判断の視点

AI出力を評価する4つの視点

1. デザイン原則の遵守

整列・近接・反復・コントラストが守られているか？視線誘導は適切か？

2. ブランド一貫性

指定したカラー・フォント・トーンが反映されているか？既存アセットとの整合性は？

3. ユーザビリティ

ボタンはタップしやすいサイズか？読みやすいコントラストか？レスポンシブは適切か？

4. 実装可能性

生成されたインタラクションは現実的か？パフォーマンスに問題はないか？

💡 Visual Architectの役割：AIが生成したものを「これで良い」と判断できるのは人間だけ

よくある失敗パターンと対策

❌ 失敗1：プロンプトが曖昧すぎる

「かっこいいサイト作って」→ AIは汎用的で特徴のないデザインを出力
✅ 対策：ターゲット、目的、避けたい印象を具体的に指定

❌ 失敗2：一度に完璧を求める

最初から細部まで指定 → AIが混乱して意図と違う出力
✅ 対策：レイアウト→機能→スタイルの順で段階的に詰める

❌ 失敗3：AI出力をそのまま採用

生成されたものを検証なしで使用 → ブランド崩壊・UX問題
✅ 対策：必ず4つの評価視点でチェックしてから承認
