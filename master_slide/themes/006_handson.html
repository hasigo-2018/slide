Gemini演習

AIでボタンバリエーションを設計する

ツール紹介：Gemini

用途：デザインの設計案・ルールをAIに考えさせる

初回アクセス：https://gemini.google.com/app

今日使う機能：

  • 構造化提案：バリエーションの種類を網羅的に洗い出す
  • カラーコード生成：HEX値を含む具体的な色指定
  • ルール言語化：状態変化のロジックを明文化
💡 AIに「全パターン出して」と頼むと、人間が見落としがちな状態も網羅できる

演習課題：ボタンバリエーション設計

目標

Geminiに「プロダクトで必要なボタンの全バリエーション」を設計させる

設計するバリエーション

  • スタイル：Primary / Secondary / Ghost / Destructive
  • 状態：Default / Hover / Active / Disabled / Loading
  • サイズ：Small / Medium / Large

アウトプット目標

各バリエーションの色（HEXコード）と変化ルールを一覧で取得

⏱️ 制作時間：10分

手順 1：バリエーション一覧を生成

Geminiに入力するプロンプト

SaaSプロダクトのボタンコンポーネントを設計してください。 【スタイルバリエーション】 - Primary（主要アクション） - Secondary（副次アクション） - Ghost（テキストのみ） - Destructive（削除・警告） 【状態バリエーション】 - Default（通常） - Hover（マウスオーバー） - Active（クリック中） - Disabled（無効） - Loading（読み込み中） 【サイズバリエーション】 - Small（補助的なアクション用） - Medium（標準） - Large（CTA用） 上記すべての組み合わせについて、以下を表形式で出力してください： 1. 背景色（HEXコード） 2. 文字色（HEXコード） 3. ボーダー（色・太さ） 4. 角丸（px） 5. パディング（px） ブランドカラーは #00d4ff（シアン）をベースにしてください。
⚠️ よくある間違い：「ボタンのデザインを考えて」だけでは曖昧すぎる

手順 2：状態変化ルールを深掘り

追加で入力するプロンプト

先ほどのボタン設計について、状態変化のルールを詳しく教えてください。 具体的に知りたいこと： 1. Hover時の色変化ルール（明度を何%上げる？） 2. Active時の色変化ルール（明度を何%下げる？） 3. Disabled時の透明度（何%？） 4. Loadingアニメーションの推奨仕様 また、以下の観点でアクセシビリティチェックもお願いします： - 背景色と文字色のコントラスト比（WCAG AA基準を満たすか） - Disabled状態でも内容が読み取れるか Figmaで実装しやすいよう、具体的な数値で回答してください。
⚠️ 「なんとなく暗く」ではなく「明度-10%」のような具体値を引き出す

Figma演習

AIの設計をコンポーネント化する

ツール紹介：Figma

用途：UIコンポーネントの作成・管理

初回アクセス：https://www.figma.com/

今日使う機能：

  • Auto Layout：パディングを自動調整
  • Component：再利用可能な部品化
  • Variants：バリエーションを1つに統合
💡 無料プランで十分使えます（3プロジェクトまで）

演習課題：ボタンコンポーネント作成

目標

Geminiで設計したバリエーションを、Figmaのコンポーネント機能で実装する

作成するもの

  • ベースボタン：Auto Layout適用済み
  • バリアント：Style × State × Size の組み合わせ
  • 合計：4スタイル × 5状態 × 3サイズ = 60パターン

制約条件

  • Geminiの出力したHEXコードを使用
  • 角丸・パディングもGeminiの指定値を適用
⏱️ 制作時間：20分

手順 1 / 4：ベースボタンを作成

操作手順

1. 長方形ツール（R）でボタンの背景を作成

2. テキストツール（T）で「ボタン」と入力

3. 両方を選択し、右クリック →「Frame selection」

4. 右パネルで「Auto Layout」をONにする（Shift + A）

Auto Layoutの設定

  • Padding：上下12px、左右24px（Mediumサイズの場合）
  • 角丸：8px（Geminiの指定値）
⚠️ よくある間違い：背景とテキストをグループ化してしまう（Auto Layoutが使えない）
✅ 正解：「Frame selection」でフレーム化する

手順 2 / 4：コンポーネント化

操作手順

1. 作成したボタンフレームを選択

2. 右クリック →「Create component」（Ctrl/Cmd + Alt + K）

3. レイヤー名を「Button」に変更

コンポーネント化の確認

左パネルのレイヤーに紫の◇マークが表示されればOK

理論編との関連

これで「Atom」が1つ完成。
この部品をコピーして使えば、すべて同じデザインが保証される。

💡 コンポーネント = 「マスター部品」。変更は自動で全インスタンスに反映

手順 3 / 4：バリアントを追加

操作手順

1. コンポーネントを選択し、右パネル「Properties」の「＋」をクリック

2.「Variant」を選択

3. プロパティ名を設定：

  • Style：Primary / Secondary / Ghost / Destructive
  • State：Default / Hover / Active / Disabled / Loading
  • Size：Small / Medium / Large

バリアント追加の方法

紫の点線枠内で「＋」ボタンをクリック → 新しいバリアントが追加される

⚠️ よくある間違い：別々のコンポーネントを60個作る（管理が破綻する）
✅ 正解：1つのコンポーネントに60バリアントを内包

手順 4 / 4：Geminiの値を適用

操作手順

1. 各バリアントを選択

2. 右パネル「Fill」でGeminiのHEXコードを入力

3. テキストの色も同様に変更

4. サイズバリアントはパディング値を変更：

  • Small：上下8px、左右16px
  • Medium：上下12px、左右24px
  • Large：上下16px、左右32px

効率化のコツ

まずPrimary × 5状態を完成させ、複製して色だけ変更すると速い

💡 Loading状態のアイコンは「⟳」や「…」で仮置きしてもOK

完成イメージ：バリアントパネル

検証演習

「1箇所修正で全部直る」を体験する

検証課題：一括修正を体験

シナリオ

「ブランドカラーを #00d4ff から #6366f1（紫）に変更したい」

やること

  • Step 1：ボタンのインスタンスを5個配置
  • Step 2：マスターコンポーネントの色を変更
  • Step 3：全インスタンスが自動で変わることを確認

確認ポイント

5個のボタンを1回の操作で変更できれば成功！

🎯 これが理論編で学んだ「一貫性が仕組みで生まれる」の正体

検証手順：一括修正の実演

Step 1：インスタンスを配置

1. 左パネル「Assets」からButtonコンポーネントをドラッグ

2. キャンバス上に5個配置（Alt + ドラッグでコピー）

3. 各インスタンスのバリアントを変更（Primary、Secondary など）

Step 2：マスターを修正

1. コンポーネント本体（◇マーク）を選択

2. Primary / Default の背景色を #6366f1 に変更

3. 関連する Hover / Active の色も調整

Step 3：結果を確認

配置した5個のインスタンスが自動で紫に変わることを確認

💡 これを60ページのサイト全体で行えると想像してください。手動なら数時間、コンポーネントなら数秒。

一括修正の効果

❌ コンポーネント未使用

✅ コンポーネント使用

実践編のまとめ

今日体験したこと

  • Geminiで全バリエーションを網羅的に設計
  • Figmaでコンポーネント化＆バリアント管理
  • 一括修正で「仕組みによる一貫性」を実感

ビジュアルアーキテクトの視点

  • AIには「網羅」を任せる - 人間が見落としがちなパターンも出る
  • 人間は「判断」に集中 - どれを採用するか、例外を許すか
  • ツールで「仕組み化」 - 一貫性は努力ではなく構造で担保
🎯 Atomic Design = 「部品を作る」のではなく「部品で考える」思考法
