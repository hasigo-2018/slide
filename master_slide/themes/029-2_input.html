Part 1
デザイントークン

値に「名前」をつけて、意味を共有する

デザイントークンとは
「意味を持った変数」

#00d4ff ではなく color-primary と呼ぶ理由

なぜ「値」ではなく「名前」で管理するのか

値で管理する場合の問題

  • 変更が困難：#00d4ffを#0088ffに変えたい → 全ファイル検索・置換
  • 意味が不明：「#1a1a2e」が背景色なのか文字色なのかわからない
  • 一貫性の欠如：人によって微妙に違う色を使ってしまう

名前で管理するメリット

  • 一箇所変更：トークン定義を変えれば全体に反映
  • 意味が明確：color-primary は「主要な色」という意味
  • 強制的な一貫性：定義されたトークン以外は使えない
💡 トークン = 「値」と「意味」のペア

トークンの3層構造

デザイントークンの4カテゴリ

システムとして定義すべきトークン

  • Color：primary, secondary, background, text, error, success...
  • Typography：font-family, font-size (xs/sm/md/lg/xl), font-weight, line-height
  • Spacing：space-xs(4px), space-sm(8px), space-md(16px), space-lg(24px)...
  • Radius / Shadow / Z-index：radius-sm/md/lg, shadow-sm/md/lg, z-modal...
⚠️ よくある間違い：すべての値をトークン化しようとする
✅ 正解：繰り返し使う値だけをトークン化する

トークン導入：Before vs After

❌ Before（値の直接指定）

問題点：値の意味が不明、変更時に全箇所修正

✅ After（トークン使用）

改善点：意味が明確、一箇所変更で全体反映

Part 2
コンポーネント階層

部品の「関係性」を Atomic Design で定義する

Atomic Design
UIを「原子→分子→有機体」で構成する

小さな部品を組み合わせて、大きなUIを作る設計思想

Atomic Design の5階層

「カート追加ボタン」はどの階層か？

結論：Atoms（原子）

カート追加ボタンは、それ以上分解できない最小単位のUIパーツです。

判断基準

  • 単一の機能：「カートに追加する」という1つの動作
  • 内部に他のコンポーネントを含まない：アイコン + テキストのみ
  • 高い再利用性：商品詳細、クイックビュー、カートページなど複数箇所で使用
💡 ボタン + 数量セレクター + 在庫表示 = Molecule（分子）
→ 複数のAtomsを組み合わせると上位階層になる

階層間の依存ルール

「この部品はどの階層？」の判断フロー

3つの質問で判断

  • Q1：それ以上分解できる？ → Yes なら上位階層、No なら Atoms
  • Q2：単独で意味を持つ？ → Yes なら Molecules以上、No なら Atoms
  • Q3：ページの一部分として独立している？ → Yes なら Organisms

実例での判断

  • 「カート追加ボタン」→ 分解不可 → Atoms ✅
  • 「数量選択 + カート追加ボタン」→ 複数Atomsの組み合わせ → Molecules
  • 「商品画像 + 価格 + 数量選択 + ボタン」→ 独立したセクション → Organisms
⚠️ よくある間違い：見た目の大きさで判断する
✅ 正解：「分解可能性」と「独立性」で判断する

Part 3
Geminiの役割

デザインシステムのドキュメント作成を自動化する

Geminiが担う
3つのドキュメント作成タスク

構成生成・命名提案・ガイドライン文章化

役割1：ドキュメント構成の自動生成

Geminiに依頼できること

デザインシステムドキュメントの目次・章立てを自動生成します。

入力例

「ECサイト向けデザインシステムのドキュメント構成を提案してください。対象はフロントエンド開発者とデザイナーです。」

出力イメージ

  • 1. 導入（目的、対象者、更新履歴）
  • 2. デザイントークン（Color、Typography、Spacing）
  • 3. コンポーネント（Atoms、Molecules、Organisms）
  • 4. パターン（フォーム、ナビゲーション、モーダル）
  • 5. 運用ルール（命名規則、更新フロー）
💡 ゼロから構成を考える手間が省ける

役割2：トークン命名規則の提案

Geminiに依頼できること

一貫性のある命名規則をトークンカテゴリごとに提案させます。

入力例

「カラートークンの命名規則を提案してください。Primitive / Semantic / Component の3層構造で、ダークモード対応も考慮してください。」

出力イメージ

  • Primitive：blue-50, blue-100, ... blue-900（明度スケール）
  • Semantic：color-primary, color-secondary, color-surface, color-on-surface
  • Component：button-bg-default, button-bg-hover, button-text
⚠️ 注意：Geminiの提案をそのまま採用せず、チームで議論して決定する

役割3：運用ガイドラインの文章化

Geminiに依頼できること

箇条書きのルールを、読みやすいガイドライン文章に変換します。

入力例

「以下のルールをデザインシステムの運用ガイドラインとして文章化してください： ・新規トークン追加は週次レビューで承認 ・命名はkebab-case ・破壊的変更はメジャーバージョンアップ」

AIが得意な変換

  • 箇条書き → 段落形式の読みやすい文章
  • 技術用語 → 非エンジニアにもわかる説明
  • ルール → 具体例付きのガイドライン
💡 文章作成の80%をAIに任せ、人間は確認・調整に集中

Gemini活用の3つの注意点

避けるべきアンチパターン

  • 丸投げ：「デザインシステムを作って」→ 文脈なしでは汎用的すぎる出力に
  • 無批判な採用：命名規則をそのまま使う → チームの慣習と合わない
  • 一度で完成を期待：1回のプロンプトで完璧を求める → 反復改善が必要

効果的な使い方

  • 下書き生成：Geminiに叩き台を作らせ、チームで磨き上げる
  • 選択肢提示：「3パターン提案して」で比較検討の材料を得る
  • 段階的精緻化：大枠 → 詳細 の順で複数回やり取りする
💡 Gemini = 「優秀なアシスタント」であり「意思決定者」ではない
