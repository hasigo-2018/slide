デザインが「崩れる」3つの原因

1. ルールの不在

「なんとなく似た色」「だいたい同じサイズ」で作ると、ページごとに微妙な差異が生まれ、統一感が失われる。

2. スケールの限界

5ページまでは「整列・近接・反復・コントラスト」で管理できても、50ページになると人間の記憶力では一貫性を保てない。

3. 協業の困難

複数人で作業すると、各人の「良いデザイン」の基準が異なり、バラバラな成果物になる。

💡 デザインシステム = ルール + 再利用可能な部品 + 拡張性

デザインの方法論

Atomic Designの5段階階層

化学の「原子→分子→有機体」の概念を借用

Brad Frost（2013年）が提唱した、UIを5つの階層で管理する方法論。

  • Atoms（原子）：最小単位（ボタン、入力欄、ラベル）
  • Molecules（分子）：原子の組み合わせ（検索フォーム = 入力欄 + ボタン）
  • Organisms（有機体）：分子の集合（ヘッダー = ロゴ + ナビ + 検索フォーム）
  • Templates（テンプレート）：ページの骨組み（ワイヤーフレーム）
  • Pages（ページ）：実際のコンテンツを入れた完成形
⚠️ よくある誤解：「Atoms = 小さい、Pages = 大きい」だけではない
✅ 正解：各階層は「再利用性」と「抽象度」が異なる

Atomic Designの階層構造

第1層：Atoms
（原子）

Atoms：これ以上分解できない最小単位

なぜ必要か

デザインの「最小公倍数」を定義することで、一貫性の基盤を作る。色が微妙に違う、ボタンのサイズがバラバラ、という問題を防ぐ。

どう作るか

  • ステップ1：デザイントークンを定義（色、フォント、スペーシング）
  • ステップ2：基本コンポーネントを作成（ボタン、入力欄、ラベル）
  • ステップ3：バリエーションを制限（例：ボタンは3種類まで）

具体例

ボタン（Primary/Secondary/Tertiary）、入力欄、チェックボックス、アイコン、テキストラベル

⚠️ よくある間違い：「小さいもの = Atoms」と勘違いする
✅ 正解：「単独で機能し、再利用される最小単位」

Atoms：実例

❌ Atomsが定義されていない

問題点：サイズ、色、角丸がバラバラ

✅ Atomsとして統一されている

改善点：サイズ、角丸、余白が統一

第2層：Molecules
（分子）

Molecules：Atomsの組み合わせで機能を作る

なぜ必要か

Atomsだけでは「ボタン」単体で終わるが、「検索フォーム = 入力欄 + ボタン」のように組み合わせることで、実用的な機能が生まれる。

どう作るか

  • ステップ1：よく一緒に使われるAtomsを特定（例：ラベル + 入力欄）
  • ステップ2：グループとして扱い、命名規則を決める（例：SearchForm）
  • ステップ3：内部のAtomsの配置ルールを定義（整列・近接の原則を適用）

具体例

検索フォーム（入力欄 + ボタン）、カード（画像 + タイトル + 本文）、フォームフィールド（ラベル + 入力欄 + エラー表示）

⚠️ よくある間違い：Moleculesに複雑すぎる機能を持たせる
✅ 正解：「単一の目的」を持つシンプルな組み合わせ

Molecules：実例（検索フォーム）

❌ Moleculesとして設計されていない

問題点：入力欄とボタンが離れている、高さが不揃い

✅ Moleculesとして設計されている

改善点：高さ統一、近接の原則で視覚的なグループ化

第3層：Organisms
（有機体）

Organisms：Moleculesの集合で「セクション」を作る

なぜ必要か

ヘッダー、フッター、商品一覧など、ページの「意味のあるまとまり」を定義する。これにより、ページ全体の一貫性が保たれる。

どう作るか

  • ステップ1：ページ内の「セクション」を特定（ヘッダー、メインコンテンツ、フッター）
  • ステップ2：各セクションに必要なMoleculesを配置
  • ステップ3：セクション全体の視覚的ヒエラルキーを設計（コントラストの原則）

具体例

ヘッダー（ロゴ + ナビ + 検索フォーム + ログインボタン）、商品カード一覧、記事リスト

⚠️ よくある間違い：OrganismsとTemplatesの境界が曖昧
✅ 正解：Organisms = 再利用可能なセクション、Templates = ページ全体のレイアウト

Organisms：実例（ヘッダー）

ヘッダーというOrganism = ロゴ + ナビ + 検索フォーム + ログインボタン

第4層：Templates
（テンプレート）

Templates：ページ全体の骨組みを定義

なぜ必要か

Organismsをどう配置するかのルールを決める。ワイヤーフレームとして機能し、複数ページに共通のレイアウトを適用できる。

どう作るか

  • ステップ1：ページの構造を決定（ヘッダー、メイン、サイドバー、フッター）
  • ステップ2：グリッドシステムを適用（12カラム、8pxグリッドなど）
  • ステップ3：レスポンシブのブレークポイントを定義

具体例

「ブログ記事ページ」「商品詳細ページ」「ランディングページ」のテンプレート

⚠️ よくある間違い：Templatesにコンテンツを入れてしまう
✅ 正解：Templatesは「枠」だけ、コンテンツはPagesで入れる

Templates：実例（ブログ記事レイアウト）

Templateは配置と余白のルールだけを定義、コンテンツは入れない

第5層：Pages
（ページ）

Pages：実際のコンテンツを入れた完成形

なぜ必要か

Templatesに実際のテキスト、画像、データを入れることで、デザインが実際に機能するかをテストできる。長い見出し、大量のテキスト、欠けた画像など、実データで問題を発見する。

どう作るか

  • ステップ1：Templateを選択
  • ステップ2：実際のコンテンツを入れる（ダミーテキストではなく本物）
  • ステップ3：エッジケースをテスト（長いタイトル、大量のタグ、画像なしなど）

具体例

「2024年10月の売上報告」というブログ記事ページ、「iPhone 15 Pro」という商品詳細ページ

💡 Pagesで問題が見つかったら、該当するAtoms/Molecules/Organismsに戻って修正

デザイントークン：
Atomsを支える基盤

デザイントークン：値に名前をつける

なぜ必要か

「#3498db」という色コードより「color-primary」という名前の方が、意図が伝わり、変更時にも一括で対応できる。

主な種類

  • カラートークン：primary, secondary, accent, text, background
  • タイポグラフィトークン：font-family, font-size, line-height, font-weight
  • スペーシングトークン：spacing-xs (4px), spacing-sm (8px), spacing-md (16px), spacing-lg (24px)
  • サイズトークン：width, height, border-radius, border-width
⚠️ よくある間違い：「blue-500」のような抽象的な名前を使う
✅ 正解：「color-primary」「color-error」のように用途を示す名前

ネーミング規則：
大規模でも崩れないルール

コンポーネントのネーミング規則

なぜ必要か

100個のコンポーネントがある時、「Button1」「Button2」では管理不能。一貫した命名規則で、誰が見ても理解できるようにする。

推奨パターン：BEM（Block Element Modifier）

  • Block：独立した意味のある塊（例：Button, Card, Header）
  • Element：Blockの一部（例：Button__icon, Card__title）
  • Modifier：状態やバリエーション（例：Button--primary, Button--disabled）

実例

Button → Button__icon → Button--primary → Button--primary--disabled

⚠️ よくある間違い：「BlueButton」「BigCard」のように見た目で命名
✅ 正解：「Button--primary」「Card--large」のように役割で命名

バリエーション管理：
制約が自由を生む

バリエーションの爆発を防ぐ

なぜ制限が必要か

「ボタンの色は何色でもOK」とすると、プロジェクトが進むにつれて無限にバリエーションが増え、一貫性が失われる。

制限の例

  • ボタン：Primary / Secondary / Tertiary の3種類まで
  • カード：Standard / Compact / Expanded の3種類まで
  • カラーパレット：Primary, Secondary, Accent, 各3色ずつ = 計9色

例外の扱い

新しいバリエーションが必要な場合は、チーム全体で議論し、デザインシステムに追加するか決定する。

💡 制約があるからこそ、デザイナーは「どう使うか」に集中できる

スケーラビリティ：大規模案件でも崩れない設計

スケーラビリティとは

5ページのサイトから500ページのサイトに拡大しても、デザインシステムが機能し続ける状態。

確保する方法

  • ドキュメント化：すべてのコンポーネントに使い方、制約、バリエーションを記載
  • バージョン管理：コンポーネントを変更する際は必ずバージョン番号を更新
  • 依存関係の明確化：「このOrganismはこのMoleculeに依存」と図示
  • 定期的な監査：使われていないコンポーネントは削除、重複は統合
⚠️ デザインシステムは「作って終わり」ではない、継続的にメンテナンスが必要

デザインシステムと4つの原則の関係

デザイン原則はシステムの基盤

Atomic Designは「部品の管理方法」、4つの原則は「部品を美しく配置する方法」。両者は補完関係にある。

  • 整列（Alignment）：Atomsの配置ルール、グリッドシステムに適用
  • 近接（Proximity）：MoleculesやOrganismsでの要素のグループ化
  • 反復（Repetition）：デザイントークンで色・フォントを繰り返す
  • コントラスト（Contrast）：Primary/Secondaryボタンの視覚的差異
💡 デザインシステムは「何を作るか」、4つの原則は「どう配置するか」

デザインシステム構築の実務手順

ステップ1：インベントリ作成（1-2日）

既存のデザインから、使われている色、フォント、ボタン、カードなどをすべてリストアップ。

ステップ2：デザイントークン定義（1-2日）

リストアップした要素を整理し、「color-primary」「spacing-md」などの名前をつける。

ステップ3：Atoms作成（2-3日）

ボタン、入力欄、ラベルなどの最小単位をFigmaで作成。バリエーションを制限。

ステップ4：Molecules/Organisms作成（3-5日）

検索フォーム、カード、ヘッダーなどを組み立て。

ステップ5：Templates/Pages作成（3-5日）

実際のページレイアウトを作成し、実データでテスト。

⏱️ 小規模プロジェクト：2週間、大規模プロジェクト：1-2ヶ月
