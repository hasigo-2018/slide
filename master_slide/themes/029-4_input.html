Section 1

なぜデザイン仕様書が必要なのか

仕様書がない現場で起きること

認知科学的な問題：「暗黙知の壁」

デザイナーの頭の中にある設計意図は、明文化しない限り他者には伝わりません。心理学では「知識の呪い（Curse of Knowledge）」と呼ばれる認知バイアスにより、作り手は「自分が知っていることは相手も分かるはず」と錯覚します。

現場で頻発する3つのトラブル

  • 実装のズレ：エンジニアがデザインの余白やカラーを目測で再現し、微妙にずれる
  • 属人化：担当デザイナーがいないと、修正や拡張の判断ができない
  • フィードバックの空転：「なんか違う」としか言えず、建設的な指示ができない
⚠️ よくある間違い：「デザインデータを渡せば仕様書は不要」→ データは「何があるか」を示すが「なぜそうしたか」は示さない

仕様書がもたらす3つの価値

仕様書なし vs あり：ディレクションの質

❌ 仕様書なし

問題点：基準がないので「感覚」でしか指示できない

✅ 仕様書あり

改善点：仕様書が「共通言語」となり、的確なフィードバックが可能に

Section 2

デザイン仕様書の構成要素
— 何を記録すべきか

仕様書を構成する3つのレイヤー

Layer 1：スタイルガイド
Style Guide

デザインの「DNA」を定義する最も基礎的なレイヤー。
色・文字・余白という3つの原子がすべてのデザインの土台になる。

スタイルガイドに記録すべき情報

カラーシステム

Primary / Secondary / Accent / Neutral の各色をHEXコードで定義し、使用比率（例：60-30-10ルール）と使用場面を明記します。

タイポグラフィ

見出し（H1〜H4）・本文・キャプションの各レベルについて、フォントファミリー、サイズ、ウェイト、行間（line-height）を数値で定義します。

スペーシング（余白体系）

4px / 8px / 16px / 24px / 32px / 48px といった基本単位を定め、要素間の余白に一貫したリズムを与えます。

⚠️ よくある間違い：「色名だけ記録する」→ 「青」ではなく「#2563EB（Primary Blue）使用場面：CTA・リンク」まで書く

Layer 2：コンポーネント仕様
Component Specification

UIの「部品」一つひとつの設計図。
Atomic Designで学んだ原子・分子・有機体の考え方がここに直結する。

コンポーネント仕様に記録すべき情報

各コンポーネントに必要な定義

  • 状態（State）：Default / Hover / Active / Disabled / Error の各見た目
  • サイズバリエーション：Small / Medium / Large の寸法・パディング
  • 内部構造：アイコンとテキストの位置関係、間隔の数値
  • 使用ルール：「Primaryボタンは1画面に1つまで」などの制約

なぜ状態の定義が重要か

ユーザーはインタラクションの中でボタンの状態変化を通じてフィードバックを受け取ります。状態が未定義だと、エンジニアが独自に判断することになり、体験の一貫性が崩れます。

⚠️ よくある間違い：Default状態だけ定義して「あとはいい感じに」→ 状態ごとの色・影・サイズの変化をすべて数値化する

ボタンコンポーネントの仕様例

Layer 3：レイアウト仕様
Layout Specification

ページ全体の「骨格」を定義するレイヤー。
ワイヤーフレームで学んだグリッドと視線誘導の理論が、ここで数値化される。

レイアウト仕様に記録すべき情報

グリッドシステム

カラム数（12カラムが標準）、ガター幅、マージン幅を定義。レスポンシブ対応のブレークポイント（例：375px / 768px / 1280px）ごとの変化も記録します。

セクション間の余白ルール

ヒーロー → 特徴セクション → CTAなど、セクション間の余白を統一的に定義。スペーシング体系（Layer 1）に基づいて整合性を保ちます。

コンテンツの優先順位

F型・Z型の視線誘導パターンに基づき、各ブレークポイントでのコンテンツ配置順序を定義します。モバイルでの並び替え（リオーダー）ルールもここに含まれます。

⚠️ よくある間違い：デスクトップだけ定義して「モバイルは適宜調整」→ ブレークポイントごとに配置・サイズの変化を明記する

Section 3

AIプラグインの動作原理
— どう自動生成されるのか

AIプラグインとは何か

Figmaプラグインの基本的な仕組み

Figmaプラグイン（Design Doc等）は、Figmaファイル内のデザインデータを自動的に読み取り、仕様書として構造化されたドキュメントを生成するツールです。人間が手作業で行っていた「デザインを見て、情報を書き出す」作業をAIが代行します。

AIが「読み取る」情報

  • レイヤー構造：要素の親子関係・グループ構造
  • スタイル属性：色（HEX/RGB）・フォント・サイズ・余白・角丸
  • コンポーネント情報：Main ComponentとInstanceの関係
  • Auto Layout設定：方向・間隔・パディング・配置
💡 ポイント：AIは「見た目」ではなく「データ構造」を読んでいる。だからFigma上の整理状態が出力品質に直結する

自動ドキュメント生成の3ステップ

Garbage In, Garbage Out の原則

AIの出力品質はFigmaファイルの整理度に比例する

AIプラグインは「賢い読み取り機」であって「優秀な推測器」ではありません。Figmaファイルが乱雑であれば、生成される仕様書も乱雑になります。これは情報理論における「ノイズ」の問題と同じです。

良質な出力を得るための前提条件

  • 命名規則の統一：レイヤー名が「Frame 387」のままだと、仕様書にも無意味な名前が入る
  • コンポーネント化：Main Componentとして登録されていない要素は、仕様書上で個別要素として扱われる
  • スタイル登録：Local StylesやVariablesに登録されていない色・文字は「バラバラの値」として出力される
⚠️ よくある間違い：「AIが自動でやってくれるから整理しなくていい」→ AIは人間の整理を自動化するのであって、整理そのものを代行するわけではない

Figmaファイルの品質と仕様書の品質

❌ 乱雑なFigmaファイル

問題点：命名未整理・コンポーネント未登録・スタイル未定義

✅ 整理されたFigmaファイル

改善点：意味ある命名・コンポーネント化・スタイル登録済み

Section 4

人間の判断が必要なポイント
— AIの限界と補完

AIが自動化できること / できないこと

AI生成仕様書のレビュー：4つの観点

観点1：正確性（Accuracy）

抽出された数値が実際のデザインデータと一致しているか。特にスタイルの「継承」や「オーバーライド」がある場合、AIが正しく読み取れていないことがあります。

観点2：完全性（Completeness）

必要な情報が漏れなく記録されているか。特にインタラクション状態（Hover/Active/Error等）は、Figma上でプロトタイプとして設定されていると見落とされやすいです。

観点3：意図の補足（Intent）

「なぜそうしたか」の設計意図が記載されているか。AIは事実しか書けないので、「Primaryボタンを青にした理由：信頼感の訴求」のような意図は人間が追記します。

観点4：可読性（Readability）

仕様書の読み手（エンジニア、別のデザイナー、クライアント）にとって分かりやすい構成になっているか。読み手に応じた粒度調整は人間の仕事です。
