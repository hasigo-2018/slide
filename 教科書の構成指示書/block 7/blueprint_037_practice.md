# 構成指示書：037 チーム開発における共同編集とコミュニケーションのルール【実践編】

## 基本情報
- **テーマ:** チーム開発における共同編集とコミュニケーションのルール
- **目的:** Block 5–6で制作したTALENTBOARDのFigmaファイルに、理論編で学んだ4本柱のルールをすべて適用し、「個人の作業ファイル」を「チームのプロジェクトファイル」に変換する
- **推奨スライド数:** 10〜12枚
- **推奨時間:** 約50分×2コマ（100分）
- **使用ツール:** Figma（ブランチ・バージョン管理・コメント機能）
- **素材:** TALENTBOARD LP（Block 5–6で制作・改善した自分のFigmaファイル）

## 重要な設計意図
- 演習は4本柱の順序通りに進める。①ファイル構造 → ②命名規則 → ③ブランチ運用 → ④コメント作法。後の柱は前の柱が整っていることが前提になるため、順序を守る
- 演習3（ブランチ運用）は「修正シナリオ」を講師が指示し、実際にブランチを切って作業→マージする一連のフローを体験させる。座学だけでは身につかない操作スキル
- 演習4（コメント作法）はペアワークで実施。受講生同士で互いのファイルにコメントを付け合い、「他者に伝わるか」を検証する。036（公開添削②）の経験がここで活きる
- 037の成果物＝「ルール適用済みのFigmaファイル」は、038の仕様書AI生成の入力として直接使用される
- 成果物は新しいデザインではなく、チーム運用に耐えるFigmaファイル。自分ではなく、次に触る人が迷わないかを基準に判断する

## ビジュアルアーキテクト育成上の位置づけ

このテーマは、題材「チーム開発における共同編集とコミュニケーションのルール」を通じて、ビジュアルアーキテクトに必要な「目的に沿って設計・判断・説明する力」を鍛える位置づけである。単に作業手順やツール操作を覚えるのではなく、このテーマの目的（Block 5–6で制作したTALENTBOARDのFigmaファイルに、理論編で学んだ4本柱のルールをすべて適用し、「個人の作業ファイル」を「チームのプロジェクトファイル」に変換する）に向けて、何を選び、なぜそうするかを自分の言葉で説明できる状態をゴールとする。

## 視覚化方針

このテーマでは、文章説明だけでなく、必要に応じてSVG図解・画像生成・既存アセットを使い分ける。特に、初学者が状況を想像しづらい箇所、Before/After、判断基準、成果物イメージ、実務での使用場面は視覚化候補として扱う。スライド生成時は「画像があると理解が速くなるか」「SVGで構造化した方がよいか」「既存素材で具体化できるか」を確認し、視覚要素が学習目的を支える場合に採用する。

## クイズ（5問・実践判断型）

| No. | 問題 | 正解のポイント |
|-----|------|---------------|
| Q1 | ファイル構造の整備で最初にやるべきことは？ | 現状把握。いきなりページを作るのではなく、今のファイルに何があるかを確認し、不要物をArchiveに退避してからテンプレートに合わせて再構成する |
| Q2 | 命名規則の適用で「Rectangle 47」が1つでも残っていると何が起きるか？ | 検索機能が使えなくなる。038でAIに仕様書を生成させる際も、レイヤー名が情報源になるため、自動生成名では「このレイヤーが何を指すか」をAIが判断できない |
| Q3 | ブランチ運用で「マージ前にレビューを通す」ルールの理由は？ | 修正内容がMainの品質基準を満たしているか、他の修正と競合していないかを確認するため。レビューなしでマージすると、上書き事故や品質低下がMainに混入する |
| Q4 | ペアワークでコメントを付ける際、032の赤入れと037のコメントの最大の違いは？ | 032は「デザインの中身」（色・配置・導線）に対する修正指示。037は「ファイル運用」（命名・構造・整理状態）に対するフィードバック。対象が違うだけで、「何を・なぜ・どう直すか」の3要素は共通 |
| Q5 | 037のルール整備が不十分な状態で038に進むと何が起きるか？ | AIに仕様書を生成させる入力の品質が低くなる。レイヤー名が「Rectangle 47」では仕様書の要素名も意味不明になり、ファイル構造が散乱していれば仕様書の対象範囲も曖昧になる。037は038の入力品質を決定する前提条件 |
