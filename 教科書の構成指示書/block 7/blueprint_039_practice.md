# 構成指示書：039 エンジニアへのハンドオフ【実践編】

## 基本情報
- **テーマ:** エンジニアへのハンドオフ
- **目的:** TALENTBOARDのハンドオフパッケージ（仕様書＋Figmaデータ＋アセット＋フォルダ構成＋実装ガイド）を実際に組み上げ、「エンジニアに渡せる状態」にする
- **推奨スライド数:** 14〜16枚
- **推奨時間:** 約50分×4コマ（200分）
- **使用ツール:** Figma（インスペクター最適化・アセット書き出し）、Gemini（実装ガイド生成補助）
- **素材:** TALENTBOARD LP（037で整備＋038で仕様書作成済みのFigmaファイル）

## 重要な設計意図
- 実践は4段階で進める。①Figmaインスペクター最適化（仕様書と並行して参照できる状態にする）→ ②アセット書き出し（全画像・アイコンを指定フォーマットで一括書き出し）→ ③パッケージ統合（仕様書＋アセット＋フォルダ構成＋実装ガイドの一式を組み上げる）→ ④ペアレビュー（「エンジニアの立場」で互いのパッケージを検証する）
- ペアレビュー（演習4）がBlock 7の最終演習。036（公開添削②）では「デザインの中身」を互いにレビューしたが、039では「渡し方の品質」をレビューする。視点が変わっても「相手の立場に立って検証する」行為は同じ
- 039の成果物＝「完成したハンドオフパッケージ」は、Block 8のポートフォリオに「実務対応力の証拠」として直接組み込まれる
- Block 7の総括として、037→038→039の一連のフローが「実務のワークフローそのもの」であることを振り返る

## クイズ（5問・実践判断型）

| No. | 問題 | 正解のポイント |
|-----|------|---------------|
| Q1 | Figmaインスペクター最適化で最も優先すべき設定は？ | カラースタイルとテキストスタイルの登録。これが未登録だとインスペクターに「#3B82F6」とHEXコードだけが表示され、エンジニアはそれが「プライマリカラー」なのか「リンクカラー」なのか判断できない。スタイル名が表示されることで、仕様書との対応が一目でわかる |
| Q2 | アセット書き出しで1x＋2xをセットで書き出す理由は？ | Retina（高解像度）ディスプレイ対応。1xだけだと高解像度画面でぼやけ、2xだけだと通常画面でファイルサイズが無駄に大きくなる。エンジニアはHTMLのsrcset属性で出し分けるため、両方が必要 |
| Q3 | ペアレビューで「readmeを読んで全体像が掴めない」と指摘された場合、何を改善すべきか？ | readmeの構成と記載順序。プロジェクト概要→ファイル構成→参照順序→ブレークポイント→フォント→注意点→連絡先の順で、エンジニアが「上から読んでいけば準備完了」になる流れにする。情報は揃っていても、順序がバラバラだと全体像が掴めない |
| Q4 | 039のペアレビューと036の公開添削②の共通点と相違点は？ | 共通点：受講生同士のレビューであり、037のコメント作法（@メンション・優先度タグ・解決済みマーク）を使う。相違点：036は「デザインの中身（設計意図・視覚品質）」を斬ったが、039は「渡し方の品質（仕様書の精度・アセットの整備度・パッケージの網羅性）」を検証する。レビューの作法は同じだが、検証の視点が異なる |
| Q5 | Block 7の3テーマ（037→038→039）の成果物が、Block 8のポートフォリオにどう組み込まれるか？ | 037のチーム運用ルール＝「チーム開発に対応できる証拠」。038の仕様書＝「設計意図を実装レベルで言語化できる証拠」。039のハンドオフパッケージ＝「デザインから実装への受け渡しを一人で完結できる証拠」。Block 8では、これらを「ビジュアルアーキテクトとしての実務対応力」の具体的なエビデンスとしてポートフォリオに組み込む |
